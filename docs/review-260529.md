# Spotykach Code & Architecture Review - 2026-05-29

Scope: a full read of `src/` plus `app.cpp`, cross-checked against the user manual. This
document records an architecture assessment, concrete issues (separated into verified and
suspected), refactoring opportunities, and feature opportunities for a fork that intends
to add missing features.

## Confidence and method

Findings are labelled:

- **[verified]** - I read the relevant code and confirmed the claim in this session.
- **[suspected]** - surfaced during exploration but not line-by-line confirmed; treat as
  a lead to check before acting. Several exploration notes turned out to be false on
  inspection (see "Corrected" below), so do not act on a [suspected] item without
  verifying.

This review is provisional. It favours flagging things worth a look over asserting
certainty. Exact line numbers are omitted because they drift; grep for the named symbols.

### Corrected during review (do NOT treat as bugs)

- The loop buffer's record pre-roll (`Buffer::set_recording`) was flagged as an integer
  underflow; it is correctly guarded by `_read_head - std::min(size_t(1), _read_head)`.
  Not a bug. **[verified]**
- Gate-out was flagged as writing both A and B every call; it is correctly branched on
  `ref == Deck::A`. Not a bug. **[verified]**
- The manual lists "gate outputs not yet implemented" and "buffer clearing not
  implemented" as limitations; both exist in this codebase (`CoreUI::_process_gate_out`,
  `Buffer::clear`). The manual is partly stale relative to this firmware. **[verified]**

## 1. Architecture assessment

### Strengths

- **Clean execution-context discipline.** The hard split - audio/DAC/timer ISRs do only
  real-time work, while SD/flash I/O and pad/MIDI handling live in the main loop - is the
  right design for this platform and is followed consistently. Large buffers are
  pre-allocated through one SDRAM pool, so the audio path does not allocate.
- **Good separation of concerns.** `Core` is hardware-agnostic and self-contained;
  `CoreUI` is the only bridge between `Hardware` and `Core`. This makes the DSP testable
  in principle and the hardware re-mappable via the `hardware.h` enums.
- **Sophisticated transport.** `SynClock`'s fractional-tick external sync with tempo
  feedback is more careful than most hobby firmware, and tap-tempo has outlier rejection.
- **Musically-aware looping.** Quantized record/loop grids, per-deck sequencer with
  parameter locks that survive mode changes, and cross-deck recording are non-trivial and
  coherent.

### Structural weaknesses

1. **`Deck` is a god class.** It carries mode-specific state (`_loop_ticks`,
   `_through_loop_ticks`, `_loop_tick_count`, `_adjust_count`, several `_is_*_queued`
   flags) that only applies in some modes, and coordinates buffer, generator, track,
   dispatcher and fx. This is the highest-risk area to change and the prime refactor
   target (see section 4). **[verified - read deck.h/deck.cpp]**
2. **Shared mutable state across contexts without synchronisation.** Parameters set in
   the main loop are read in the audio callback. Today this is mostly safe because writes
   are word-sized scalars, but it is implicit and fragile; any future multi-word
   parameter (e.g. a struct snapshot) will need an explicit publish mechanism (double-
   buffer + single index swap). **[suspected - no memory model is documented]**
3. **Magic numbers scattered outside `config.h`.** Tempo/grid scaling constants
   (e.g. the `720000` tick factor in `Deck`), the increment-smoothing coefficient in
   `Generator`, swing scaling in `Divider`, and several blink/timeout durations in the UI
   are inline literals. `config.h` already exists as the home for these. **[verified for
   several; representative, not exhaustive]**
4. **No automated tests and no host build.** Everything is verified on hardware. The
   DSP and timing classes (`Tempo`, `Divider`, `SynClock`, `SpeedMap`, `Config` parser,
   `wav` header) are pure logic and could be unit-tested off-target. This is the single
   biggest leverage point for safely adding features. **[verified - no test target]**
5. **Filename/spelling inconsistencies** (`time.iterval.h`, `read_settigs`,
   `_next_inetrval`) are harmless but signal the absence of review tooling. **[verified]**

## 2. Verified issues

| # | Issue | Severity | Location |
|---|-------|----------|----------|
| V1 | `Config::_is_loaded` has no initializer and the constructor does not set it. If `config.txt` is missing/empty, `is_loaded()` reads an indeterminate value. | Medium | `config.h` (`Config()` and `bool _is_loaded;`) |
| V2 | `DeckStorage::next_tape`/`previous_tape` wrap using `kStorageSlotCount` instead of `kStorageTapeCount`. Both are 6 today so it works, but it is the wrong constant and will break silently if tapes != slots. | Low (latent) | `storage.cpp` |
| V3 | `_srcBuf3` (a full 42 s stereo source buffer, ~16 MB SDRAM) is allocated but never handed out - `sourceBuffer()` is only called twice (decks A and B). Either dead memory or a reserved-but-unused feature hook. | Medium (waste) / opportunity | `buffer.sdram.cpp`, `core.cpp` |
| V4 | Open TODO: granular parameter override order is undefined ("knob-over-cv-over-MIDI-over-event"). Affects predictability when multiple sources drive the same grain parameter. | Low | `generator.cpp` |
| V5 | `Ws2812` has a TODO'd, unhandled DMA error path; a stalled transfer has no recovery. | Low | `ws2812.cpp` |

V3 is the most interesting: ~16 MB of SDRAM is reserved with no consumer. If you want a
feature that needs a per-deck scratch/undo/second buffer, the allocation already exists;
if not, removing it frees significant SDRAM.

## 3. Suspected issues (verify before acting)

These came out of exploration and are plausible but unconfirmed. Listed so they are not
lost; each needs a targeted read.

- **S1 - Divider triplet drift.** Triplet timing multiplies the next-trigger index by
  `0.6666...f` and truncates to integer, which can accumulate phase error over long runs.
  Check whether it re-derives from an absolute grid each bar (which would bound the
  error) or accumulates. `divider.cpp`.
- **S2 - Vox decay start at high speed.** The grain decay-start appears to be computed as
  `size - slope*increment`; at high playback increments this could go negative and cut
  grains early. `vox.cpp`. The manual's note that "Slice-mode overdubbing may glitch when
  synced to external MIDI clock" may be related.
- **S3 - External-sync boundary.** A `>= ticks_per_clock - 1` comparison in `SynClock`
  for 24-PPQN MIDI into a 48-PPQN grid is an off-by-one candidate; worth a focused trace.
- **S4 - Calibrator blocks the main loop.** `Calibrator` collects CV samples with
  `System::Delay` in a loop (hundreds of ms). It runs only during explicit calibration,
  so audio continues (separate ISR) but the UI freezes. Acceptable, but confirm it cannot
  be entered accidentally. `calibrator.cpp`.
- **S5 - Config parser robustness.** Fixed 8-byte property/line buffers and
  `memcmp(..., 8)` matching mean long or malformed lines are silently mishandled, and
  whitespace is stripped mid-token. Init-only and low-risk, but brittle. `config.h`.
- **S6 - FatFS reentrancy.** If any future change calls a `Card` method from the audio
  context, FatFS is not reentrant. Today it is main-loop only. Keep it that way.

## 4. Refactoring opportunities

Ordered by value-to-risk for someone about to add features.

1. **Stand up an off-target test harness.** Compile the pure-logic classes (`Tempo`,
   `Divider`, `SynClock`, `SpeedMap`, `Config::fill`, `wav_header`, `Follower`,
   `XFade`/`OnePoleSmoother`) for the host with a tiny test runner. This pays for itself
   immediately because the timing/parsing bugs above become reproducible. Note: the
   project Makefile has no `test` target today; this would be a new, host-only build.
2. **Extract mode behaviour out of `Deck`.** The Reel/Slice/Drift differences are
   currently `switch`/flag soup. A small strategy object (or at least a `struct
   ModeProfile` of the per-mode settings applied in `_set_mode`) would isolate mode logic
   and make a fourth mode tractable.
3. **Centralise constants.** Move the inline tempo/grid/smoothing/blink literals into
   `config.h` (or a new `timing.h`) with names and unit comments. Many are already
   documented there; finish the job.
4. **Formalise the param-publish boundary.** Document (and where needed enforce) that
   core parameters are scalar and written only from the main loop. If any compound
   parameter is added, introduce a double-buffered snapshot rather than field-by-field
   writes.
5. **Tidy naming.** Rename `time.iterval.h` -> `time.interval.h`, `read_settigs` ->
   `read_settings`, `_next_inetrval` -> `_next_interval`. Mechanical, low risk, improves
   greppability.
6. **WS2812 error handling.** Close the TODO with a timeout + resync so a dropped DMA
   does not freeze LED updates.

## 5. Feature opportunities

Grounded in the manual's own "Known Limitations / Not implemented yet" plus natural
extensions. Each notes what already exists in code as a starting point.

### From the manual's stated gaps

- **F1 - User-facing buffer clear.** The manual lists "buffer clearing function" as not
  implemented. `Buffer::clear()` exists but is only called internally on re-arm. Adding a
  dedicated gesture (e.g. a long-press combination) to clear a deck's audio without
  re-recording is low-risk and directly requested. **[verified clear() exists]**
- **F2 - Spot pad functionality.** The manual says the Spot pad is "planned for a future
  update." In code the `P::Spot` case is minimal (calibration entry). It is a free
  control for a new global function (e.g. snapshot/recall, freeze, panic, tap variant).
- **F3 - Recorded-material persistence / explicit save-all.** The manual lists
  persistence between power cycles as not implemented, but a preload/auto-load path
  exists in `Storage`. Clarify the actual behaviour and consider an explicit "save both
  decks" gesture and a session snapshot. **[verified preload exists]**
- **F4 - Confirm/finish gate outputs.** Gate-out is implemented (`_process_gate_out`,
  7 ms pulse on grain re-trigger) though the manual calls it unimplemented. Verify timing
  on hardware and document it; potentially add per-mode gate behaviour (e.g. clock-out vs
  trigger-out selection). **[verified code exists]**

### Reusing the spare 42 s buffer (V3)

- **F5 - Per-deck undo / A-B take.** The third source buffer is already allocated. A
  one-level undo of the last overdub, or an A/B "take" toggle per deck, fits the existing
  memory budget without new allocation. Requires routing one source buffer to a deck on
  demand and a copy strategy (copy is not RT-safe in-callback; stage it in the main loop).

### Natural extensions

- **F6 - MIDI CC parameter control.** MIDI currently handles clock + note triggers +
  start/stop only. Mapping CCs to the main knobs (start/size/pitch/mix/fx) would make the
  instrument far more usable in a MIDI rig, and the `Config` text parser already provides
  a place to declare mappings.
- **F7 - Modulation routing matrix.** Modulators are hard-wired to fixed targets. A small
  per-deck source->destination selection (even a fixed menu) would multiply the
  expressiveness with modest code.
- **F8 - Additional FX or FX routing.** The `Fx` host cleanly supports adding effects
  modelled on `Drive`/`Reduce`. Candidates: a filter (the `Biquad` already exists), a
  reverb, or serial-vs-parallel Grit/Flux routing.
- **F9 - More LFO shapes / per-source assignment.** `LFO` already abstracts waveform
  type; exposing the full set per source (rather than the manual's fixed A=S&H/square,
  B=sine/saw split) is mostly UI work.

## 6. Suggested first steps for this fork

1. Add the host test harness (refactor #1) and lock down `Tempo`/`Divider`/`Config`
   behaviour, then revisit S1-S3 and S5 with tests.
2. Fix V1 and V2 (trivial, correctness).
3. Decide V3: either reclaim `_srcBuf3`'s 16 MB or commit to F5 (undo/take) that uses it.
4. Pick one user-visible feature from F1/F2/F6 - all are well-scoped and align with the
   manual's roadmap.
