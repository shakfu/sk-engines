# Changelog

All notable changes to the sk-engines firmware are recorded here.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/). This project does not yet expose a version constant in the firmware; entries are grouped by the Git tag / release they ship in.

## [Unreleased]

### Added

- **glitch engine (`ENGINE=glitch`) — a dual-deck lo-fi / circuit-bent noise voice, host-tested (hardware pending).** `make ENGINE=glitch` gives each deck (A/B) one of **12 curated algorithms** ported from Rob Scape's [Noisferatu](https://github.com/rob-scape/noisferatu) (upstream `firmware/Noisferatu/algos.h`), blended by the crossfader and placed by the routing switch — a pair of glitch oscillators you mix. The point is the **digital-glitch aesthetic** the platform's cleaner engines don't cover (the Plaits-based [mosc](docs/engines/mosc.md), the Rings-based [reso](docs/engines/reso.md) out-class plain noise): **buffer glitch** (sparse-chunk playback, a wandering grain window, and SoundScaper-style **address bit-mangling**), **logic-noise** (Benjolin/Atari-punk tri-XOR / square-NAND / inharmonic-FM / ring-mod), **generative scale blips** (Phrygian / pentatonic / Bernoulli enveloped triangles), and **rhythmic noise** (dust, clock-divided bursts) — curated from Noisferatu's ~50 entry points, which are mostly preset variations of ~12 distinct ideas. Per deck: **Alt+PITCH** scrolls the algorithm (ring-dot selector), **SIZE/POS** are the two algorithm-specific params, **PITCH** is a ±2-octave master pitch, **ENV** is an output tone (low-pass tilt — the naive oscillators alias by design), **MIX** is volume, the **Play pad** regenerates the glitch buffer, and the **Routing** switch sets the stereo image (A|B / mono / random-pan). The DSP is a clean reimplementation, not a textual include of the sketch: **de-Arduino'd** (`<cstdint>` + the sketch's own xorshift PRNG replace `<Arduino.h>`/`random()`), **de-globaled** into a per-instance `glitch::Voice` (the original's ~246 file-scope globals would have made the two decks share state — a multi-instance bug; the decks now carry distinct PRNG seeds so the same algorithm sounds different on each), and **retuned from 16 kHz to 48 kHz** (phase increments derived from the injected rate so pitches are correct rather than 3× sharp; envelope decay coefficients rescaled to preserve the author's wall-clock times). A normal **SRAM** build that is fully **self-contained** — no SDRAM arena, no SD/streaming, no transport; the two ~8 KB glitch buffers live in the engine object (~79% SRAM_EXEC). Host-tested (`make -C host test-glitch`: every algorithm finite/bounded/non-silent, plus the control surface — algorithm select + readback, both params, master pitch, ENV tone, Play-pad regen, routing, and deck independence). User doc: [`docs/engines/glitch.md`](docs/engines/glitch.md); the curation rationale, the port, and the buffer-silence fix in [`docs/dev/glitch-impl.md`](docs/dev/glitch-impl.md). **License:** Noisferatu is GPLv3, so the port inherits it — `src/engine/glitch/glitch_voice.h` and `src/engine/glitch/` (and any `ENGINE=glitch` binary) are **GPLv3**, not MIT like the rest of the project (the GPLv3 files carry SPDX headers and the license text + rationale live in `src/engine/glitch/{LICENSE,NOTICE.md}`); no other engine includes the glitch voice. (Not yet hardware-verified — the by-ear retuning of the per-algorithm param ranges and buffer density is the open item; the dropped Vinyl-Crackle algo needs a PROGMEM sample table not present in the source.)

- **qdelay engine (`ENGINE=qdelay`) — a dub/ambient flavor of the delay, host-tested + hardware-verified.** `make engine-qdelay` is a second take on the [dual delay](docs/engines/delay.md) inspired by Tiago Lobato Gimenes' [qdelay](https://github.com/tiagolr/qdelay) plug-in: it keeps the delay engine's entire control grammar (tempo-synced musical divisions, feedback, mix, PITCH ±1-octave transpose, ENV feedback tone, the MODFREQ/MOD_AMT mod LFO, the Freeze/Reverse pad gestures, and the Stereo/DoubleMono/Ping-pong topologies) and swaps only the **character palette** from Clean/Tape/Shimmer to **Clean / Diffuse / Duck** on the Reel/Slice/Drift switch. **Diffuse** routes the per-tap feedback through a shared stereo **8-stage allpass diffuser** so repeats progressively smear into a dense, reverb-like dub wash (SIZE sets the smear depth); **Duck** runs a per-tap fast-attack/slow-release peak follower over the input and attenuates the heard wet while you play, so the repeats stay out of the way and bloom in the gaps (the feedback loop itself is not ducked, so the tail keeps building and is revealed when the input drops). The diffuser is the new header-only `src/dsp/diffuser.h`, a **JUCE-free port** of qdelay's `Diffusor` (a TARON MiniVerb-style cascade, L/R coefficient tables detuned for width) with **no heap in the audio path** — `capacity_floats(sr)` sizes a single block and `init(mem, sr)` sub-allocates every stage from it, so its ~235 KB of delay lines live in the SDRAM arena (alongside the two ~6 s delay buffers) and SRAM is untouched. A normal **SRAM build** (links at ~77% SRAM_EXEC at `-O2`, comfortable headroom). The diffuser is a single stereo instance keyed off deck A's SIZE, applied per-tap only when that deck is in Diffuse (so DoubleMono can mix a Diffuse deck with a Clean/Duck deck) and skipped entirely when neither deck is. Why a separate engine rather than new delay characters: the binding hardware constraint is the control surface (a few knobs + two 3-way switches), not code/CPU, so qdelay can't expose the desktop plug-in's ~40 params; it picks the highest-value, lowest-cost slice (feedback diffusion + ducking) and maps it onto the existing mode switch as a distinct flavor. Host-tested (`make -C host test-qdelay` covers ring-out/feedback, the three distinct+bounded characters, Duck collapsing the steady wet energy ~1500→~0 then ringing out after input stops, ping-pong vs DoubleMono isolation, Reverse, and max-feedback stability; plus a standalone `make -C host test-diffuser` for the diffuser module — impulse smear, exact `clear()`, size response, L/R decorrelation, null-mem passthrough). User doc: [`docs/engines/qdelay.md`](docs/engines/qdelay.md); internals/deltas in [`docs/dev/qdelay-impl.md`](docs/dev/qdelay-impl.md). **License:** the diffuser is a derivative of qdelay's GPLv3 `Diffusor`, so `src/dsp/diffuser.h` + `src/engine/qdelay/` (and any `ENGINE=qdelay` binary) are **GPLv3**, not MIT like the rest of the project — the GPLv3 files carry SPDX headers and the license text + rationale live in `src/engine/qdelay/{LICENSE,NOTICE.md}`; no other engine links the diffuser. (Flashed on the H7 — boots and the effect was heard, including the Diffuse wash and Duck gestures; per-deck independent diffusers and a pre/post diffusion option are scoped as follow-ons gated by free controls.)

- **Delay engine: Reverse on the Rev pad.** The `delay` engine's previously-stubbed Rev play-pad gesture now toggles a per-deck **Reverse** that reads the buffer backwards over a delay-length window. `Tap::read_rev(win)` walks a phase forward `0..win` while the read offset (`win − phase`) shrinks from the oldest sample toward the write head, with two heads half a window apart raised-cosine crossfaded so the offset wrap is click-free (the same seamless-wrap trick the pitch `Shifter` uses); the forward and reverse reads share a new linear-interpolated `read_buf(off)` helper. Reverse composes with all three characters and topologies (it only changes the read) and the mod LFO still wobbles the window; the play-pad LED shows cyan when reversed (white = frozen, green = input level), and the gesture is idempotent (toggling twice is bit-identical to forward). Host-tested (`make -C host test-delay` gains a reverse group: differs from forward, rings out, bounded, toggles back exactly) and hardware-verified (flashed on the H7; the reverse read is click-free across delay lengths). The diffuse/ducking ideas noted as "still open" in the delay dev notes were taken up in the new `qdelay` flavor instead of as delay characters. ([`docs/engines/delay.md`](docs/engines/delay.md), [`docs/dev/delay-impl.md`](docs/dev/delay-impl.md))

- **softcut engine (`ENGINE=softcut`) — a dual-deck crossfaded overdub looper on monome's [softcut-lib](https://github.com/monome/softcut-lib), hardware-verified.** `make ENGINE=softcut` gives four `softcut::Voice`s (two per deck, a Rev-swapped pair like [shuttle](docs/engines/shuttle.md)). Unlike the buffer-tape engines, softcut's read/write head plays **and** records the same loop at once, with subsample-accurate click-free loop crossfades and interpolated **overdub** — the sound-on-sound that shuttle/tape can't do. It uses **record-defines-loop** semantics: on an empty voice the first **Alt+Play** records a fresh take and the second closes the loop at exactly the length you played; a voice that already has content overdubs on top. Controls: **PITCH** = rate (bipolar, noon=stop, CCW=reverse, snaps to unity on engage), **POS/SIZE** = loop window, **MIX** = volume, **ENV** = overdub feedback (1=infinite layering → 0=overwrite), **Alt+PITCH** = SD slot, **Alt+POS** = pan, **MOD_AMT/MODFREQ** = loop crossfade time / rate slew, the **FLUX pad** = a sweepable post filter (+PITCH cutoff, +MIX reso); **Seq** realigns all voices with a click-free `cutToPos`, and the **Routing** switch pans the voices (manual / auto-spread / random). **SD load** (boot preload + Alt+PITCH slot select, float32-mono WAV, like shuttle) and **SD save** in two modes — **Alt+Seq** saves the full take (lossless), **Alt+Rev** saves the trimmed POS/SIZE loop ("bounce what you hear") — to the selected slot (and **holding Alt+Seq erases the focused voice**, a clean discard that writes nothing), so loops round-trip across power cycles. A normal **SRAM** build (the softcut DSP is ~11 KB; the full firmware links at ~89% SRAM_EXEC at `-O2`); 4 voices is the on-device CPU budget (~62% avg / 79% peak worst case, measured), with per-voice 10.9 s (2^19-frame) buffers in the SDRAM arena. The vendored softcut core (`src/engine/softcut/vendor`) carries five embedded-targeted port fixes (a 3 MB-per-voice debug-buffer stub, a 64-bit→32-bit atomic to avoid a missing libcall, an `<iostream>` strip that was the difference between fitting and overflowing SRAM_EXEC, FadeCurves init-order member initializers caught by ASan, and `-DM_PI/-DM_PI_2`). Host-tested (`make -C host test-softcut`). User doc: [`docs/engines/softcut.md`](docs/engines/softcut.md); the feasibility spike + on-device CPU measurements in [`docs/dev/softcut-spike.md`](docs/dev/softcut-spike.md) and the internals/bring-up in [`docs/dev/softcut-impl.md`](docs/dev/softcut-impl.md). (FX beyond softcut's built-ins, a 6-voice build, and phase-quantised sync are scoped as follow-ons.)

### Changed

- **CMake build (`CMakeLists.txt` + `Makefile.cmake`) brought to full parity with the Makefile.** The opt-in CMake path was WIP (14 of 20 engines) and actually broken for *every* engine; it now builds the **same 20 engines** as the canonical Makefile, verified by building each. Three parts: **(1) Every missing engine registered** — the SRAM engines `glitch`/`qdelay`/`softcut` and, newly, the **QSPI-execute engines `mosc`/`csound`/`chuck`** that were Make-only. The QSPI support reuses the insight that `BOOT_SRAM` and `BOOT_QSPI` both define `BOOT_APP` (only the linker script differs), so it adds a per-engine `ENGINE_LDSCRIPT` (`alt_sram.lds` default; `alt_qspi.lds` for mosc/csound, `alt_qspi_chuck.lds` for chuck), external static-lib linking (`libcsound.a`/`libchuck.a`), the `-u _printf_float` + SDRAM-pool `--wrap=malloc,…` link options, and chuck's per-TU `-fexceptions/-frtti` (the rest of the firmware stays `-fno-*`). **(2) Two pre-existing CMake breakages fixed** that prevented any engine from linking: `src/version.cpp` was omitted from the source list *and* `SPK_VERSION_STR`/`SPK_ENGINE_STR` were never defined (so every link failed on undefined `firmware_banner()`); and the `reso` block still pointed at the old `reso/thirdparty/stmlib` path after stmlib was relocated to the shared `common/thirdparty/stmlib`. **(3) The ~5% larger `SRAM_EXEC` gap vs the Makefile closed to ~0.65%** (root-caused by section→symbol→linker-map diffing): a **fork gap** — the fork sets `-DDSY_DISABLE_USB_HOST` in libDaisy's *Makefile* but not its CMake build, so `sys/fatfs.cpp`'s `USBH_Driver` reference dragged the entire unused USB-host-MSC + USB-device-CDC + Logger stack (~8 KB) into the link via a global-handle static initializer `--gc-sections` cannot drop — fixed with `target_compile_definitions(daisy PRIVATE DSY_DISABLE_USB_HOST)` (removes only USB-MSC *host*, not SD-card FatFs, so streaming engines are unaffected); plus FatFs being built as a CMake sub-library at `-O3` vs the Make build compiling `ff.c` at the app opt (`-O2`/`-Os`), fixed with `target_compile_options(FatFs PRIVATE ${APP_OPT})`. The residual ~0.65% is a benign newlib `impure_data` (full-`libc` reentrancy struct vs nano). The root `Makefile` stays the canonical, hardware-proven build; CMake remains the opt-in alternative (`make -f Makefile.cmake ENGINE=<name> build`).

- **gigaverb (`ENGINE=gigaverb`): SIZE knob now slews `roomsize` instead of writing it raw.** gigaverb's `roomsize` scales the lengths of the reverb tank's delay lines, so a raw write made the delay taps jump and crackle on every knob step (and ADC jitter passed straight through) — audibly noisy while turning SIZE. The value is now stored as a target in `set_param` and ramped toward it once per audio block by a new optional `W::tick()` hook on `GenEngine` (detected via SFINAE, so other gen wraps are unaffected and the call compiles away), with a ~40 ms one-pole glide that snaps on the first set (no boot ramp). Only `roomsize` is slewed; every other gigaverb param still writes through directly. Links at ~83% SRAM_EXEC (the slew is byte-neutral). (`src/engine/gigaverb/gigaverb_engine.h`, `src/engine/gen/gen_engine.h`)

- **Project licensing is now MIT with documented GPLv3 exceptions.** The repo was uniformly MIT; two new engines introduce GPLv3 code, so the licensing posture is now explicit in a `## License` section in [`README.md`](README.md). **`qdelay`** brings `src/dsp/diffuser.h` (derived from qdelay's GPLv3 `Diffusor`), so that file + `src/engine/qdelay/` (and any `ENGINE=qdelay` binary) are **GPLv3**. **`glitch`** ports the algorithms of the GPLv3 [Noisferatu](https://github.com/rob-scape/noisferatu) in `src/engine/glitch/glitch_voice.h`, so `src/engine/glitch/` (and any `ENGINE=glitch` binary) is **GPLv3**. Each carries the full GPLv3 text at `src/engine/<name>/LICENSE` with the rationale + attribution in `src/engine/<name>/NOTICE.md`, and the GPLv3 source files carry `SPDX-License-Identifier: GPL-3.0-only` headers. No other engine links/includes the GPLv3 code, so every other build stays MIT. The qdelay reference plug-in was only ever a local, untracked working copy (never vendored into the repo) and is not needed to build; it has been removed.

### Removed

- **Reverb: the parked Freeverb voice (`freeverb.dsp` + `faust_kernel_freeverb.h`) is deleted.** It was kept as a lighter swap-back alternative to the third reverb voice but was never wired into the engine (the three active voices are Dattorro plate / Zita hall / Greyhole) and wasn't liked, so it's removed. No build or behaviour change — nothing referenced it. The standalone gen~ [gigaverb](docs/engines/gigaverb.md) engine is untouched; reverb docs (`docs/engines/reverb.md`, `docs/dev/reverb-impl.md`) and the Dattorro/Greyhole source comments drop their Freeverb mentions.

### Fixed

- **Host test suite builds under Apple clang 21 (macOS Tahoe).** The Sequoia→Tahoe Xcode update bumped Apple clang to version 21, whose stricter front-end broke `make -C host test` at three pre-existing spots (the firmware ARM build via `arm-none-eabi-g++` was unaffected, so this was host-only): `src/dsp/adenv.h` used `size_t` without `#include <cstddef>` (it had relied on transitive include order); `src/config.h`'s `Config::fill` declared `auto line_size = 8` and used it as an array bound — a non-const VLA clang 21 rejects in a constant-expression, now `constexpr`; and `host/Makefile` still pointed `RINGS_SRC` + the include path at `reso/thirdparty/stmlib`, which had been relocated to the shared `common/thirdparty/stmlib` (a stale path that broke `test-reso`, surfaced only once the clang errors cleared). All 22 host suites pass again (`make -C host test`).

## [0.5.1]

### Added

- **Real ChucK `MidiIn` on the bare-metal build (prototype; build-verified, on-hardware wake test pending).** `libchuck.a` is now compiled with ChucK's `MidiIn`/`MidiOut` device classes enabled (`__DISABLE_MIDI__` dropped from all three ChucK define sites — `scripts/fetch_chuck.sh`, the firmware `Makefile`, and `pod/Makefile.chuck`, kept ABI-consistent), but with `midiio_rtmidi`'s RtMidi backend stripped: there is no OS MIDI API or callback thread on bare metal, and `rtmidi.cpp` (166 KB) stays out. A new `MidiInManager::inject()` feeds bytes straight into ChucK's per-VM MIDI buffer + event-wake path, so a `.ck` patch can use the desktop-portable `MidiIn min; min.recv(msg);` idiom (`examples/chuck/midi_in.ck`) instead of host-specific globals. The **full channel-voice MIDI stream** is forwarded (real velocity, NoteOff, CC, pitch-bend, aftertouch, program change) plus system realtime (clock/start/continue/stop): a new `IEngine::handle_midi_message(status, d1, d2)` virtual (no-op for engines that only use `handle_midi_note`) is driven from the UI MIDI poll (`core.ui.midi.cpp`) for every representable event; `ChuckEngine` overrides it to enqueue raw bytes on a lock-free ring (`chuck_midi_in.h`), and `ChuckEngine::process()` drains it and injects into device 0 on the run() thread, right before `ck->run()`, so a shred blocked on `min => now` is woken that block. `ChuckEngine::handle_midi_note` is kept only to map channel→deck for the panel gate-in LED. The vendored-ChucK edits live in `scripts/patches/midi_daisy.patch`, applied idempotently by `scripts/fetch_chuck.sh` (because `thirdparty/chuck` is gitignored and regenerated). The firmware (`make engine-chuck`) links with matching ABI. Ported from the `daisy-apps` sibling project; replication guide in [`docs/dev/chuck-midi-in-porting.md`](docs/dev/chuck-midi-in-porting.md), and the user-facing capabilities + remaining incompatibilities (no `MidiOut`, single virtual device / no enumeration, 3-byte messages only) are documented in [`docs/engines/chuck.md`](docs/engines/chuck.md). (SysEx / system-common aren't representable in a 3-byte `MidiMsg` and are not forwarded; on-hardware wake test still pending.)

- **mosc engine (`ENGINE=mosc`) — a dual macro-oscillator on the Mutable Instruments [Plaits](https://github.com/pichenettes/eurorack/tree/master/plaits) DSP, hardware-verified.** `make engine-mosc` gives each deck a complete `plaits::Voice` — the **full 24-engine** Plaits synthesis voice (virtual analog, waveshaping, FM, wavetable, grain, additive, chord, speech, swarm, noise, particle, string, modal, drums, plus the newer "engine2" models: VA-VCF, phase distortion, 6-op FM, wave terrain, string machine, chiptune) — so the two decks are two independent oscillators. Knobs map Plaits' whole panel: PITCH = note, **Alt+PITCH = engine select (24 models)**, SIZE = harmonics, POS = timbre, ENV = morph, MOD_AMT = LPG decay, MODFREQ = LPG colour, MIX = level. The per-deck **Mode** switch picks **Gate** (each trigger strikes the LPG/decay envelope — Play pad / gate-in / MIDI / Seq) vs **Drone** (LPG bypassed, engine runs open), and the global **Routing** switch combines the two voices three ways: **Stereo** (A→L, B→R), **DoubleMono** (mono sum), and **GenerativeStereo** (an out/aux spread that uses Plaits' otherwise-discarded auxiliary output for width). The three per-deck CV jacks are wired: V/Oct → note, CV_SIZE_POS → harmonics, CV_MIX → timbre. Like csound/chuck it is a **`BOOT_QSPI`** build (the 24-engine voice is ~292 KB of code, too big for the 186 KB SRAM_EXEC, so it executes from QSPI flash) — but **unlike** them it still synthesises from the platform SDRAM arena (a normal arena engine that merely *executes* from QSPI), so no SD card or external library is needed; the Plaits DSP is vendored in-tree (`src/engine/mosc/thirdparty/plaits`). Cloned from the [reso](docs/engines/reso.md) engine (same author, same `stmlib`, same PIMPL + arena-placement pattern). User doc: [`docs/engines/mosc.md`](docs/engines/mosc.md); internals, the QSPI build, and the 24-engine order in [`docs/dev/mosc-impl.md`](docs/dev/mosc-impl.md).

- **Shared `stmlib` for the Mutable Instruments engines (reso + mosc).** Both engines depend on Émilie Gillet's `stmlib` support library; the two vendored copies were byte-for-byte identical (same upstream, unchanged for years), so they are now a single trimmed copy at `src/engine/common/thirdparty/stmlib` (the union of the Rings and Plaits closures, 19 files), shared via the `STMLIB_TP`/`STMLIB_INC` make vars. reso's former `src/engine/reso/thirdparty/stmlib` was removed (its tree is now `rings` only); this is a dedup with no functional change.

## [0.5.0]

### Added

- **ChucK engine (`ENGINE=chuck`) brought up and running on hardware.** `make engine-chuck` runs a full [ChucK](https://chuck.stanford.edu) VM behind `IEngine`: the engine creates a `ChucK` instance, compiles a built-in `.ck` program (a `SawOsc -> LPF -> dac` drone reading `speedA`/`mixA`/`sizeA` globals), runs the synthesis VM in the audio callback, and applies live panel input through ChucK's global-variable queue (`setGlobalFloat`). Like Csound it is a **`BOOT_QSPI`** build (libchuck's ~1.1 MB of code can't fit SRAM, so it executes from QSPI flash) with ChucK's STL/heap-heavy allocations routed to a free-capable SDRAM pool (the platform heap stays in SRAM). The static library is the WebChucK source subset + feature defines (no MIDI/network/threads/dlopen), cross-built from upstream `ccrma/chuck` (pinned `chuck-1.5.5.8`) by `scripts/fetch_chuck.sh` with a small shim sysroot standing in for the POSIX/libsndfile userland ChucK assumes. **Verified end-to-end on a bare Daisy Pod** (audible drone + working knobs; confirmed over SWD running in `Chuck_UGen::system_tick` / `Chuck_VM_Shreduler::advance` / `Chuck_Globals_Manager::handle_global_queue_messages`). The full bring-up - four independent root causes found by attaching a debugger, and the SWD flash/inspect tooling - is in [`docs/dev/chuck-pod-poc.md`](docs/dev/chuck-pod-poc.md); roadmap in [`docs/dev/chuck-impl.md`](docs/dev/chuck-impl.md).

- **ChucK SD `.ck` patch bank + Alt+PITCH live switching, hardware-verified on the spotykach.** A card of `chuck/0.ck`..`7.ck` programs turns the engine into a runtime-loadable synth: at boot it auto-loads the lowest-numbered slot (the built-in drone if the card has none); **hold Alt and turn PITCH** to scroll a per-patch ring selector and **release** to switch live; anything missing/empty/uncompilable falls back to the built-in so the unit always sounds. The live swap runs on **one persistent ChucK VM that is never torn down**: a swap stops the current patch's shreds (`removeAllShreds()` + a one-frame `run()` flush, so the old UGens detach from `dac` before the next patch sporks), then either **re-sporks the target patch's cached compiled `Chuck_VM_Code`** or, on the patch's first visit, compiles it once and pins the bytecode. This is the fix for a cumulative patch-swap **memory leak**: ChucK 1.5.5.8 never frees its built-in type system on VM teardown (`Chuck_Env::cleanup()` frees only the base types behind a `// TODO`; `~Chuck_Context` keeps its namespace on success), so *every* tear-down-and-reload strategy leaked — recompile-per-swap, `CK_MSG_CLEARVM` reset, and full VM destroy+recreate (the worst: it re-leaked the entire multi-MB type system every swap until the SDRAM pool exhausted and previously-fine patches began overrunning). With one VM + a compile-once cache, total compiles are bounded by the number of *distinct* patches, so memory rises once per patch then stays flat across unlimited cycling — confirmed on the cased unit by ear and by a new on-panel **SDRAM-pool meter** (`METER=1` build, ring B = live pool usage; ring A = CPU load). A always-on **CPU-overrun safeguard** mutes a patch whose `run()` can't keep up (solid-red rings) so the controls stay alive and you can Alt+PITCH away. The patch bank scan/quantizer is host-tested (`make -C host test-chuck-patch`). User doc: [`docs/engines/chuck.md`](docs/engines/chuck.md); the full root-cause + fix writeup and the roadmap are in [`docs/dev/chuck-impl.md`](docs/dev/chuck-impl.md). (Sound-file UGens — `SndBuf`/`WvIn`/`WvOut` — remain unavailable; a planned WAV-over-FatFs bridge for `SndBuf` is scoped as M6 in the dev doc.)

- **More `csound` example orchestras (`examples/csound/3..6.csd`).** Four new ready-to-copy patches exercising a broader slice of the core opcode set than the original drone / super-saw / poly-lead trio: **`3.csd`** a resonant "acid" voice (a self-oscillating `moogladder` ladder filter with an `lfo`-swept cutoff), **`4.csd`** a stereo reverb pad (drone + MIDI through a shared bus into `reverbsc`, with a `tanh`-limited dry/wet output), **`5.csd`** a dub echo (a feedback delay line via `delayr`/`delayw` + a movable `deltapi` tap), and **`6.csd`** a self-playing generative line (`randomh` sample-and-hold picking semitones on each `metro` tick). All follow the established conventions (the `speedA`/`sizeA`/… control channels, `instr 1` + `instr MidiNote`, the global-bus + limiter pattern for the effects) and use only core, table-less opcodes; all seven (0–6) pass desktop `csound --syntax-check-only`. (`examples/csound/README.md`)

### Fixed

- **ChucK no longer aborts/HardFaults at boot - four root causes (the engine was previously linking but dying on hardware with a dark LED).** None were ChucK-on-Cortex-M incompatibilities; the platform, bootloader, linker script, and SDRAM were always fine. (1) **`abort()` in `compileCode()`** - `--specs=nano.specs` ships libstdc++ **without exceptions**, so every `std::__throw_*` is a bare `bl abort`; ChucK stringifies float literals via `std::to_string` -> `vsnprintf("%f")`, which without float-printf returns a negative length -> `std::string(buf, buf-1)` -> `__throw_length_error` -> `abort`. Fixed by linking **`-u _printf_float`** (added to the `ENGINE=chuck` branch; was already in the csound branch). (2) **HardFault right after init** - the VM was **never `start()`ed**, so `ChucK::globals()` returned `NULL` until `run()` lazily auto-started the VM *inside the audio ISR*, racing the main loop. Fixed by calling **`_ck->start()`** in `init()` (single-threaded, before audio) and null-guarding `globals()`. (3) **Imprecise BusFault in the SDRAM allocator** - the pool (`CsoundPool`, shared by both QSPI engines) is **not reentrant**, and `setGlobalFloat`'s `new` on the main loop raced `ck->run()`'s allocations in the audio ISR, corrupting the free list. Fixed with a **PRIMASK critical section** around every pool op (`chuck_alloc.cpp`). (4) **Dead/noisy knobs** - the bare Pod harness called `set_param` in an unthrottled loop, flooding ChucK's 16384-slot global queue until the VM choked draining it (~100% CPU, main loop starved); reading every block then exposed ADC jitter as zipper noise. Fixed (Pod harness only) by reading knobs **once per audio block in the callback** with a **deadband + one-pole smooth**; the spotykach UI is already event-driven (`PotMoved` -> `_apply`), so it needs no equivalent. Debugging used new in-firmware crash capture (overrides of `abort`/`__assert_func` plus `g_chuck_init_stage`/`g_chuck_init_error` buffers, readable over SWD) and a `pod/daisy_qspi.cfg` OpenOCD config that flashes (`stmqspi` IS25LP064A) and attaches over ST-Link without DFU. All four fixes are in the spotykach `ENGINE=chuck` firmware, now **hardware-verified on the cased unit** (it boots, runs the VM, and makes sound). (`docs/dev/chuck-pod-poc.md`)

## [0.3.2]

### Added

- **Generated Faust engines can now use both decks (`dfilter`, `voice`).** The Faust generator (`scripts/gen_faust_engine.py`) previously emitted only single-deck engines - `FaustEngine::set_param` discarded the deck index, so deck A and deck B knobs collapsed onto one control set (chorus). A manifest `deck_mode` key now selects two ways to use **both** decks as independent banks. **`parallel`** (DoubleMono): two instances of one **mono** kernel, deck A on the left channel and deck B on the right, each with its own knobs, on `FaustEngine<Traits>` with a new `decks` dimension (`decks==1` keeps the single-deck path byte-for-byte). **`series`** (chain): **two different** kernels wired A->B with deck A driving stage 1 and deck B stage 2 - covering FX->FX and instrument->FX - on a sibling `FaustChainEngine<Traits>` template (`src/engine/faust/faust_chain.h`); the stages get engine-scoped namespaces (`fx_<engine>_<stage>`) so stage names can't collide. Both advertise `CapDualDeck` and need no platform change (the platform already delivers every knob for both decks; `set_param` just honours the deck arg). Shipped with two demos: **`dfilter`** (parallel - a resonant low-pass per channel, ~83% SRAM_EXEC) and **`voice`** (series - a drone oscillator into a resonant filter, the instrument->FX case that also exercises the 0-input chain path, ~84%). Each is `.dsp`(s) + a JSON manifest with no hand-written C++; the generator emits the wrapper, wires the build (`Makefile`/`engine_select.h`/`CMakeLists.txt` marker blocks), registers the kernel(s), and emits the control-diagram spec (deck B drawn as a `deckB` override for the series case). (`docs/engine-types/faust.md`, `docs/engines/{dfilter,voice}.md`)

- **Dual virtual RadioMusic (`radio` engine).** `make ENGINE=radio` builds a streaming sample player modelled on Music Thing Modular's [RadioMusic](https://github.com/TomWhitwell/RadioMusic), as a **dual** instrument: each deck (A/B) is an independent virtual radio browsing one shared SD library of numbered banks (`/radio/0`..`/radio/15`), blended by the crossfader and placed by the routing switch. Its signature is the **free-running virtual playhead** - a per-deck monotonic frame clock advances every block, and tuning to a station seeks its file to `(clock + START) mod length` before streaming forward, so every station sounds as though it kept broadcasting while you were tuned elsewhere (leave one and return and it has advanced). Per deck: **PITCH** = station tuning (+ V/oct CV), **POS** = start offset (+ CV), **SIZE** = 0.5–2x varispeed, **ENV** = inter-station static (a filtered-noise "tuning hiss" crossfaded in on a switch), **MIX** = volume, **Alt+PITCH** = bank (held ring-dot selector), the mix fader blends A/B, the routing switch sets the stereo topology, and the **Play pad / gate-in** RESET a stopped deck. It reuses the `tape` engine's `SPK_USE_STREAM` streaming stack (lock-free per-deck SDRAM ring + main-loop FatFs pump) and adds a headerless raw-16-bit codec (`RawStreamReader`), a directory scan, and the playhead - so stations stream off SD with no in-RAM length cap (the original recordings run ~25 min / 130 MB; only a tiny per-station name+length index sits in RAM). Stations are **signed 16-bit mono PCM** at 48 kHz: a headerless `.raw` (the original RadioMusic format, length = filesize/2) assumes 48 kHz unless an optional one-line `/radio/rate.txt` rebases the resampler, so an unconverted 44.1 kHz card plays at correct pitch. `scripts/convert_radio_audio.py` converts or whole-card-`mirror`s a stock RadioMusic card (resampling 44.1→48 kHz, preserving names). `capabilities() = CapOwnDisplay | CapDualDeck | CapAux`. Host-tested (`make -C host test-radio`: the playhead modulo, station/bank quantization, varispeed, static, the codec) and verified on hardware; ~83% SRAM_EXEC. (`Makefile.cmake`, `docs/engines/radio.md`, `docs/dev/radio-impl.md`)

- **`radio` SD robustness: macOS metadata filter + anti-stutter guards.** Reading an original RadioMusic card copied via Finder, the bank scan indexed the hidden AppleDouble companions (`._NAME.raw`, ~4 KB of metadata macOS writes next to every `NAME.raw` on a FAT card) as bogus tiny "stations" that looped a sliver of garbage - a fast stutter, interleaved one-for-one with the real stations as you tuned. `scan_bank` now drops them three ways: the FAT hidden/system attribute (`AM_HID|AM_SYS`, which macOS sets on dot-prefixed files), a leading-dot name, and a 32 KB minimum size; the converter skips dotfiles on the source side too. Three further guards stop any re-open-driven stutter, since a (re)open flushes and re-seeds the per-deck ring: **station-select hysteresis** (a deadband so CV/pot noise near a station boundary can't chatter the target), **settle-every-prepare** (a target oscillating on a boundary never settles, so the engine holds the last clean station instead of re-opening ~5x/s there), and a **debounced RESET that is a no-op on a live deck** (a repeated/floating-gate trigger can't flush a playing stream). Verified on hardware (the stutter was field-reported and is resolved). (`src/hw/stream_deck.cpp`, `src/engine/radio/radio_engine.{h,cpp}`, `scripts/convert_radio_audio.py`)

- **`radio`: 16-bit `.wav` stations with per-file sample rate.** Banks can now mix headerless `.raw` and **16-bit-mono PCM `.wav`** stations. A WAV body is int16 like a `.raw`, so they share one streaming path: `RawStreamReader::begin_wav` parses the header (a chunk walk validating `fmt=1`/16-bit/mono) for the body offset+length and the file's **own sample rate**, so a 44.1 kHz `.wav` plays at correct pitch with **no `rate.txt`** (the rate ratio is per-deck - the WAV header rate, or the global `rate.txt` for raw). Indexing a `.wav` costs one `f_open`+header-parse per file at bank load (a `.raw` stays a pure `f_stat`); playback cost is identical. The converter gains `--format wav`. Host-tested (header parse / seek / rewind / rate + rejection of stereo/8-bit/float, and the engine dispatching to the WAV path with the header rate rebasing playback).

- **`make diagrams`: render the d2 diagram sources to SVG.** The d2 sources moved to `docs/diagrams/` (sources) with their rendered SVGs in `docs/media/` (output), and a new `make diagrams` target renders each `docs/diagrams/*.d2` → `docs/media/<name>.svg` via the `d2` CLI - an incremental pattern rule (a diagram re-renders only when its source changes), with a clear error if `d2` isn't installed.

The version reaches the compiler as `-DSPK_VERSION_STR` from a Makefile `SPK_VERSION` (omitting `--dirty` so it only changes per commit, not per edit); a `build/.version-stamp` (mirroring the existing `.engine-stamp`) forces just `version.o` + `app.o` to recompile when it changes. The banner literal is anchored against `-Wl,--gc-sections` (a `used` attribute plus a log-independent reference in boot code) so it survives in release builds where nothing logs it.

- **Delay engine gains characters, topologies, modulation, and freeze.** Beyond the tempo-synced clean delay, the `delay` engine now has: a **feedback tone** control on **ENV** (a one-pole low-pass in the loop - up = open/clean, down = darker dub repeats); three **characters** on the **Reel/Slice/Drift switch** (`ConfigId::Mode`, per deck) - **Clean** (bit-for-bit as before), **Tape** (a wow/flutter LFO on the read time + the tone low-pass + soft saturation), **Shimmer** (the feedback pitch-shifted +12 each pass into an octave wash); three **topologies** on the **route switch** (`ConfigId::Route`) - **DoubleMono** (two independent mono delays, as before), **Stereo** (linked, a coherent stereo delay), and **Ping-pong** (linked + cross-feedback, echoes bouncing L↔R); a **modulation LFO** on the previously-unused **MODFREQ** (rate, `set_mod_speed`) + **MOD_AMT** (depth, `ModAmp`) that modulates the delay time for chorus/flange/vibrato (Tape's wow is the same LFO with a floor); and **Freeze** on the **Play pad** (`on_play_pad`, per deck) - loops the buffer at unity feedback while the input still passes dry, so you can play over a held loop. The inline pitch shifter was refactored into a reusable `Shifter` (output transpose + shimmer feedback), and `Tap::process` split into `read_color()`/`write_out()` so ping-pong can cross the two taps' feedback; Tape/Shimmer soft-clip the loop for stability while Clean stays bit-for-bit. Host-tested (`make -C host test-delay`, 7 groups: ring-out + feedback length, distinct/finite characters, ping-pong cross-feed vs DoubleMono isolation, ENV tone, max-feedback stability, modulation, freeze-sustains-vs-decay). Builds at ~80% SRAM_EXEC. (`docs/engines/delay.md`)

- **Route-aware reverb: DoubleMono runs two mono plates (heavy voices stay single).** The `reverb` engine now reads the panel **routing switch** (`ConfigId::Route`) and adapts at runtime: **Stereo / GenerativeStereo** = one stereo voice (deck A's selection) over the stereo pair (deck B's strip inert); **DoubleMono** = an independent **mono PLATE per deck** - deck A reverberates the left input into the left output, deck B the right, each driven by its own knob bank, channel-isolated (a silent input stays silent, host-verified). A full voice set is allocated **per deck** up front (`_rv[DeckRef::Count][kReverbCount]`, ~2.1 MB Faust / ~6.2 MB with gigaverb, trivial against the 48 MB arena), so the route is a runtime branch in `process()` with no reallocation. `capabilities()` gains `CapDualDeck`; `route()` reports topology and the three mode L/C/R LEDs show the route.

  **The DoubleMono cap (plate-only).** The Zita hall and gen~ gigaverb are heavy and SDRAM-bound; running two at once thrashes the H7 D-cache. Measured on hardware via the METER build: one hall ~18.6%, **two halls ~70% avg / ~89% peak** (super-linear ~3.8x - memory-latency-bound, which the host bench couldn't model), two plates ~41% / ~61%. So DoubleMono is capped to **plate-only** (`eff_voice()` forces plate; the heavy voices are stereo-route-only and can never run two-up), keeping the worst case at two plates (~61% peak) with comfortable headroom. The mode-switch selection is remembered across route changes. (`docs/engines/reverb.md`)

- **Reverb algorithm selection moved to the Reel/Slice/Drift mode switch.** The plate/hall (/gigaverb) selector is now the dedicated **3-position mode switch** (`ConfigId::Mode`) - one tactile position per algorithm - replacing the old held **Alt+PITCH** (`CapAux`) knob gesture. The switch only takes effect in a stereo route (deck A selects the voice); in DoubleMono the voice is forced to plate (see the cap above). Switching is a pointer flip that re-applies the deck's cached knob values to the newly-active kernel (no stranded knob). `CapAux` and the held-selector ring render are gone. Also: **Decay moved to the PITCH knob** (the biggest knob, the most impactful character control), swapping with POS which now drives Tone.

- **Optional gen~ gigaverb as a third reverb voice.** `make ENGINE=reverb REVERB_GIGAVERB=1` folds Tom Erbe's gigaverb (the [gen~ export](docs/engines/gigaverb.md)) into the reverb engine as a third selectable voice. It is wrapped as a non-Faust `GigaverbVoice` (driving the gen~ export by parameter index), allocates its gen~ state from a dedicated ~2 MB arena slice **per deck** (two independent gen~ instances are safe - genlib data objects are per-allocation, no global name registry), and synthesizes a true wet/dry crossfade in software (pins gen~ `dry` to 0, blends against the input) so its Mix knob matches the Faust voices. Pulls the gen~ runtime in (~9 KB code, SRAM_EXEC ~97%); plain `make ENGINE=reverb` stays a lean two-voice build.

- **Reverb host test + cost benchmark + on-device CPU meter toggle.** `make -C host test-reverb` exercises the reverb through the public `IEngine` surface (param round-trip, Mode/Route change semantics, both algorithms ringing out, every knob role bound, the mode switch swapping the kernel, the DoubleMono plate-only cap, and channel isolation); `make -C host bench-reverb` times `process()` for both topologies. A global `make ... METER=1` opt-in enables the existing `Meter::cpu` `CpuLoadMeter` for any engine, writing `load% max/avg/min` to the external USB CDC every ~250 ms via a **direct non-blocking transmit** (drops if the host isn't draining) - so the meter can never hang the main loop the way the daisy Logger's blocking `TransmitSync` does.

- **Buffer-based bipolar/reverse varispeed tape engine (`shuttle`).** `make ENGINE=shuttle` builds the random-access counterpart to the streaming `tape` engine: audio lives in SDRAM (not SD-streamed), so the read pointer is random-access and **reverse**, **freeze**, and **loop windowing** are trivial - the trade is a finite, RAM-capped length (currently 30 s/track). It runs **four independent mono tape tracks** - two per deck (A/B) - which free-run at their own speeds and sum into the stereo bus. Each track is a contiguous `float32` buffer carved from the SDRAM arena (1.44 M frames / 5.76 MB at 30 s/48 kHz; 4 tracks = 23 MB of the 48 MB arena), played by a **signed fractional read pointer** (`double`); reverse is a negative speed, freeze is speed 0 -> silence, ends wrap. `CapOwnDisplay | CapDualDeck | CapAux | CapAltPos | CapPitchPickup`. Host-verified (10 test groups), builds + links on target (~82% `SRAM_EXEC`), pending hardware bring-up. (`docs/engines/shuttle.md`, `docs/dev/shuttle_impl.md`)

- **Shuttle PITCH = bipolar capstan speed; Play snaps to unity.** PITCH is a per-track bipolar speed knob: a deadzone about **noon = stop (silence)**, clockwise = forward to **+2x**, counter-clockwise = **reverse to -2x**. Unity (+1x) lands off-centre, so the **Play pad snaps** the focused track to +1x and arms a one-shot pickup re-seed so the absolute pot must be swept across unity before it retakes the speed.

- **Shuttle four-track focus (Rev swap) + global re-align (Seq pad).** Each deck holds two tracks; all four play at once, but a deck's knobs (PITCH/POS/SIZE/MIX/Alt+PITCH) and ring address only the **focused** track. Tapping a deck's **Rev pad** swaps the focus (edrums mechanism: the backgrounded track keeps playing; a one-shot re-seed catches the pots without jumps). The four tracks drift out of phase by design; the **Seq pad re-aligns all four** to their loop start in one atomic gesture, declicked per audibly-rolling track (a ~1.3 ms fade-out -> jump -> fade-in, applied in the ISR off a flag so it is race-free and all four snap on the same block).

- **Shuttle loop window (POS / SIZE).** Each track plays a window `[start, start+length)` of its buffer rather than the whole recording: **SIZE** sets the length (full buffer down to a `kMinLoopFrames` stutter floor), **POS** slides the start across the uncovered tail (the window always stays inside the recording; at SIZE = full, POS is inert). The varispeed pointer wraps within the window both directions, and the Seq re-align snaps to the window start. Both are per-track (repoint on a Rev swap).

- **Shuttle audio source: record + SD load.** A track fills its buffer by **recording** the live input (Alt+Play, overwrite from the start) or by **loading** an existing `/shuttle/` WAV from the card into RAM (Alt+PITCH selects the slot), drained over a few `prepare()` passes; the drain advances by `play_consume`'s actual returned byte count (zero-fill is destination-only), so an underrun is re-pulled, never baked in as silence. The shuttle keeps its files in its **own `/shuttle/` folder** (filenames `tape_<a|b>_<n>.wav`), isolated from the streaming `tape` engine's `/tapes/` so the two engines never share or overwrite each other's takes.

- **Shuttle boot preload (all four tracks).** On power-up the shuttle fills **all four tracks** from the card if their files exist - track *t* of each deck loads slot *t*'s file (deck A: `tape_a_1/_2.wav` -> tracks 0/1; deck B: `tape_b_1/_2.wav`) - so a freshly powered unit is ready to jam without a manual Alt+PITCH load (the RAM engine can't stream lazily the way `tape` does). Per-track conditional (an absent slot file leaves that track empty) and, since a deck's two tracks share one SD stream, **serialized per deck** (the two decks load in parallel). The card mounts cooperatively early in boot, so the load waits across `prepare()` passes until the volume is up, then is abandoned after a boot deadline if there is no card / no files.

- **Shuttle per-track pan + routing switch.** The four tracks now pan **individually** into the stereo bus - previously each deck summed its two tracks to mono before a single deck-level pan, so the two tracks of a deck could only share one pan position. The panel **routing switch** (`ConfigId::Route`, shown on the mode L/C/R LED) picks where each track's pan comes from: **LEFT** (DoubleMono) = the manual per-track Alt+POS pan; **CENTRE** (Stereo) = auto-spread, a deck's two tracks fanned hard L/R for instant stereo width; **RIGHT** (GenerativeStereo) = a random equal-power pan per track (LCG, no platform RNG), rolled **once on entry** - unlike the tape engine, shuttle guards the roll on the actual switch transition because the platform calls `set_config` every loop. Alt+POS became a per-track knob (was per-deck), so the platform now also re-seeds `ParamId::AltPos` on a Rev focus swap to keep the pot pickup from jumping.

- **Shuttle idle ring reads as "on", with a per-track hue.** A loaded-but-stopped deck previously painted its ring black (looked powered-off); it now breathes a dim standby glow (a slow raised-cosine pulse off the engine's own clock) in the **focused track's hue** - track 1 indigo, track 2 amber - so a glance tells you which of a deck's two tracks is in focus. Live transport keeps its direction colors (record red / forward green / reverse cyan / frozen dim-white), and the Rev-swap flash now uses the newly-focused track's hue so the color change itself signals the swap.

- **`CapPitchPickup`: route PITCH through the pickup-gated path.** A new capability so an engine that snaps the speed (shuttle's Play->unity) can hold the snap via `take_param_reseed` - the absolute PITCH pot must be swept across the snapped value before it retakes control. Engines without it keep raw (un-gated) PITCH byte-for-byte.

- **Headless shuttle test.** `make -C host test-shuttle` runs `host/test_shuttle.cpp` (10 groups, via a controllable `FakeClock` for the pad debounce and a per-deck `FakeStream` for the SD-load drain): bipolar speed map, record + unity playback, reverse-wrap + silence-at-noon + loop, Play->unity snap (one re-seed), Rev swap + track independence, SD load, Seq re-align (all four atomic), realign declick (no large output step across the jump), the POS/SIZE loop window, and the boot preload (fills all four tracks from their slot files, serialized per deck, skipping a track whose file is absent).

- **Audio-prep script + docs for the streaming format.** Both streaming engines load only mono 32-bit-float WAV at 48 kHz, so `scripts/convert_tape_audio.py` batch-converts arbitrary sources (via ffmpeg or sox), verifies each output header against the firmware's requirement, and can name files straight into deck slots (`--engine tape|shuttle`, `--deck a|b`, `--start-slot`); it warns past the shuttle's ~30 s RAM cap. `docs/preparing-audio.md` documents the format and the ffmpeg/sox one-liners, linked from the tape and shuttle engine docs.

- **Tape: post-FX resonant low-pass filter on the grit pad, soft-limited bus.** Each tape deck gains a 2-pole resonant low-pass after the wow/flutter + Jiles-Atherton chain - a dub/tape tone control. It rides the **grit knob-edit gesture** rather than dedicated knobs: hold **grit + PITCH** = cutoff (exp, ~40 Hz .. 20 kHz), hold **grit + MIX** = resonance (Q 0.7 .. ~10, squared so the low half of the knob stays gentle), reusing the platform's existing `GritIntensity`/`GritMix` params - the tape engine just reinterprets them, so there is **no CoreUI/platform change**. Cutoff boots **open** so the filter is inert until swept down (its pickup is seeded open -> turn down to engage). The summed two-deck output bus is then **soft-limited** (`daisysp::SoftLimit`, the cubic soft-clip the edrums/granular buses use) so the resonant peak and two simultaneous decks can't clip the codec. Host-verified (low cutoff attenuates a 6 kHz tone, resonance boosts near the corner, limiter caps the peak at ~0 dBFS); pending hardware bring-up.

- **CMake build at parity with the Makefile.** The opt-in CMake path now resolves `reverb`, `shuttle`, and `gigaverb` (the gen~ engine's full define/include/warning set), and the `Makefile.cmake` frontend gains the matching `engine-*` flash targets plus the host-side `faust-gen`/`gen-engines`/`test-scripts` targets. All three configure and compile; `gigaverb` linked end-to-end. The Make build stays canonical.

### Changed

- **Edrums voices are now tweakable live (grit/flux macros + per-drum gain).** The reworked synth voices were static because the sequencer owned every knob; sound was fixed per model. Now the **SOS knob is per-drum gain** (the previously-missing mix/level control), **decay moved to `grit+SOS`**, and the **grit/flux pad modifiers expose four per-drum timbre macros** - `grit+PITCH`=drive, `flux+PITCH`=pitch-sweep amount, `flux+SOS`=body/noise balance, `flux+POS`=noise brightness - each a **bipolar offset on the model baseline** (noon = the model unchanged, so a fresh kit sounds identical until tweaked) addressing the focused drum and repointing on a Rev swap. That gives eight live sound axes per drum (pitch, gain, decay, drive, sweep, brightness, body/noise, model), all four drums independent. It rides the platform's existing grit/flux held-modifier layer (`GritIntensity`/`GritMix`/`FluxIntensity`/`FluxMix`/`FluxFb`, all pickup-seeded), so **no platform change** - decay sits on `grit+SOS` rather than `Alt+SOS` precisely because the `Feedback` channel the latter routes to is not seeded. Host-checked (`make -C host test-edrums`: SOS=gain incl. silence at 0 and louder with more, decay-on-`grit+SOS` lengthens the tail, all timbre macros finite/bounded when pushed). Not yet exposed: an independent body-vs-noise snap decay (needs a sixth modifier channel) and saving tweaked kits as presets.

- **Edrums synthesized voices reworked for depth and realism.** The `edrums` `Voice` gained, per model: **independent body and noise amp envelopes** (`noise_decay`) so the snap and the tonal ring decay on their own timescales (the biggest realism limiter before - one shared envelope); a **per-model pitch-envelope time** (`pitch_ms`, replacing the fixed ~25 ms) so the kick gets a long ~55 ms drop and the hat ~5 ms; a **high-passed hat** (the noise filter is now a runtime-typed `BiquadSection` switched band-pass/high-pass, vs the old dull narrow band-pass) for a metallic "tss"; an optional **second detuned body partial** (`partial_ratio`/`partial_mix`) for snare/tom richness; gentle **body saturation** (`drive` -> soft-clip) and an **attack click** (`click`, kick beater / tom mallet); and a per-model **`base_scale`** so each drum tunes to its own range under the one shared PITCH knob (e.g. a ~60 Hz kick despite the platform's 0.5 default). All five voices are now defined by a single `kModels` table - the one surface to tune by ear. Host-checked (`make -C host test-edrums`, a per-model group: every model renders finite/bounded/audible, and the high-passed hat measures far brighter than the kick's sine body). Builds at ~84% SRAM_EXEC; the voicing constants are first-principles starting points pending an on-hardware listen.

- **Tape wow/flutter rate favors very low frequencies.** The MODFREQ -> rate map was linear with too high a floor (`0.5 .. 2.5 Hz` wow / `6 .. 12 Hz` flutter), so even the lowest setting felt too fast. It is now a **cubic** curve with a low floor (`0.1 .. 2.5 Hz` wow / `0.5 .. 12 Hz` flutter): the bottom of the knob is a slow drift (~10 s period) and the rate climbs to a reasonable wobble only near the top. Endpoints (top speed) are unchanged.

- **gen~ engine build targets drop the `gen_` prefix.** The gigaverb engine renames `gen_gigaverb` -> `gigaverb` (`ENGINE=gigaverb`, `src/engine/gigaverb/`, `SPK_ENGINE_GIGAVERB`), and the generator convention changes to match: `scripts/gen_engine.py` now wires a gen~ export as a plain `ENGINE=<name>` / `src/engine/<name>/` / `SPK_ENGINE_<NAME>` (no `gen_`/`SPK_ENGINE_GEN_`), so a gen~ engine sits beside native and Faust engines with no prefix - and the only `gen`-named directory is the shared infra `src/engine/gen/`. The gen~ provenance now lives in the engine's docs + `manifest.json`, not its target name.

- **SD streaming service gated by capability (`SPK_USE_STREAM`), not engine name.** The streaming platform (`StreamDeck`/`FatFile`, SDRAM rings, keep-card-mounted) was guarded by `#if defined(SPK_ENGINE_TAPE)`; it now keys on a Makefile-set `SPK_USE_STREAM` flag that both `tape` and `shuttle` define. The `app.cpp` construct/pump/inject of `_stream` is the one shared touch point (a hardware service the platform must own); the rest now reference the capability, so the next streaming engine is a one-line Makefile change. Non-streaming engines stay byte-identical (`tape` output unchanged - both macros are defined for a tape build).

### Fixed

- **Streaming WAV load: validate format + tolerate metadata chunks (was silent distortion / no-load).** The shared `WavStreamReader` (tape + shuttle playback/load) handed a file's raw body bytes straight into the engine's mono-float frames with no format check, so a 16-bit, 32-bit-**integer**, or stereo WAV was reinterpreted bit-for-bit as float32 - extreme full-scale distortion (a small int sample becomes a ~1e13 float). It also parsed only the first 64 header bytes, so an externally-authored file whose `data` chunk sits behind a `fact`/`LIST` metadata block (e.g. ffmpeg/DAW float output lands it at offset 90-128) silently failed to load. `begin()` now (a) **walks the RIFF chunk list by seeking** - skipping any `fact`/`LIST`/`JUNK` metadata of any size with no fixed offset ceiling, and handling `WAVE_FORMAT_EXTENSIBLE` - to locate `data`, exactly as a conformant reader must, and (b) rejects anything that is not the native record format - `AudioFormat`/`BitsPerSample` per the build (float32 default, int16 under `LOFI_INT16`), mono, 48 kHz - returning false so `start_play` raises the deck's error flash instead of playing garbage. The format checks are engine-capability gates (the stream path does no conversion), not WAV-spec strictness. Device-recorded tapes (canonical mono-float 44-byte header) are unaffected. New host tests cover the rejects (16-bit / int / stereo / non-48k), the native accept, and parsing `data` behind `fact`/`LIST` metadata at any offset (incl. the real ffmpeg/Logic layout, a chunk before `fmt `, and odd-sized chunks).

- **Tape FX boots OFF (was a colored, un-reducible default).** The per-deck wow/flutter + Jiles-Atherton FX seeded its four knobs at non-zero defaults (drive/char/wow 0.3, rate 0.4), so a freshly loaded tape played with audible wow/flutter and saturation. Worse, the platform seeds each knob's pickup from those values, so a pot physically below the seed could never be swept down across it (soft-takeover) - the wow/flutter literally could not be turned off (it read as a "record needle moving between grooves"). `_fx_n` now defaults to all-zero per deck: the FX boots neutral and every param is freely reducible from zero. (`src/engine/tape/tape_engine.h`)

- **Audio-prep script writes the canonical WAV header itself (was noise on playback).** `scripts/convert_tape_audio.py` let ffmpeg/sox author the WAV container, which for 32-bit float **always** carries a `fact` chunk (and a DAW source's `LIST`/INFO metadata rode along), pushing the `data` body to offset 58-128 instead of the canonical 44 - all spec-valid, but a firmware reader that keys on the canonical offset then played that metadata as audio (structured noise). The script now decodes to **raw float** (`ffmpeg -f f32le` / `sox -t f32`) and writes a byte-exact copy of the firmware's own 44-byte header (`data` at 44, no `fact`/`LIST`), so output plays on any reader regardless of how its header parser works. Pairs with the chunk-walking reader above: either fix alone resolves the noise.

- **Tape: a wrong-format file now flashes distinctly from an empty slot.** A rejected load **strobes** the deck's amber error indicator (~4.5 Hz) when the file exists but was refused for its format, versus the **steady** amber of a missing/empty slot - so "convert this file" is distinguishable from "nothing here" on the panel. The cause is detected via `IStreamDeck::exists` (present-but-refused = format).

- **CMake `tape` build was unlinkable (missing `SPK_USE_STREAM`).** The CMake `tape` branch defined only `SPK_ENGINE_TAPE`, but the SD-streaming TUs (`stream_deck`/`fat_file`/`buffer.sdram`) gate their bodies on `SPK_USE_STREAM`, so they compiled empty and the link failed on undefined `StreamDeck`/`FatFile` symbols. The branch now sets `SPK_USE_STREAM`, matching the Makefile.

## [0.2.3]

### Added

- **Dual streaming tape-deck engine (`tape`).** `make engine-tape` builds two independent **mono** tape decks (A/B) that stream record/playback to/from the SD card, removing the in-SDRAM loop-length cap (~42 s float / ~84 s int16, `kSourceMaxSeconds`) - each deck plays/records an arbitrarily long file. A deck is play-XOR-record (no overdub), so the two run like a pair of record decks (play A while recording B, then play both and beat-match by ear). The engine is thin: in the audio ISR it only moves float frames between the platform's lock-free per-deck SDRAM rings and the buffers; the slow FatFs I/O runs in a dual-deck `StreamDeck` pump on the main loop, reached via a per-deck `IStreamDeck*` injected through `EngineContext`. Native mono float WAV (no conversion); one 1 MB ring per deck + a 32 KB shared scratch; all guarded by `#if defined(SPK_ENGINE_TAPE)` so other engines stay byte-identical. `CapOwnDisplay | CapDualDeck | CapAux | CapAltPos`; NOT `CapTapeStorage`. Dual-deck core hardware-verified; ~81.5% `SRAM_EXEC`. See `docs/engines/tape.md`, `docs/dev/tape-impl.md`.

- **Tape transport + per-deck controls.** Per deck: **Play** = play, **Alt+Play** = record (300 ms debounce; Rev pad reserved for reverse); **PITCH** = varispeed (`exp2((v-0.5)*2)`, 0.5x..2x, +/-1 octave); **MIX** = playback volume. Two MONO inputs (deck A records input A, deck B input B - never summed) mix to a stereo bus driving the headphone + the individual jacks. Display per deck: idle ring off / green = play / red = record / amber = rejected start, on the Play-pad LED + ring.

- **Tape routing + pan.** The **routing switch** (`Route`) picks the topology - LEFT = each deck panned by its **Alt+POS** knob, CENTRE = both centered (mono sum), RIGHT = random pan - shown on the mode L/C/R LED. The **mix fader** (`Crossfade`) blends deck A/B (DJ-style: centre = both full, ends = one only).

- **Tape loop modes (ENV knob).** Per deck, by quadrant: none / plain loop / faded loop (~50 ms seam de-click) / Frippertronics (decays ~0.6x per pass, then auto-stops). Looping lives in the stream (`PlayStream` rewinds the WAV at EOF); the fade/decay is engine-side via `loop_frames` + source-frame tracking, with a Frippertronics fade-out flagged from the ISR and stopped in `prepare()`. (`src/memory/{audio_stream,wav_stream}.h`)

- **Tape slots - non-destructive takes.** 8 slots per deck under `/tapes/` (`tape_a_<n>.wav` / `tape_b_<n>.wav`), selected by **Alt+PITCH** (the ring shows the 8-slot selector while Alt is held); the next Play/record targets the active slot. The selector distinguishes **recorded vs empty** slots (selected bright / recorded mid / empty dim), probed via a new `IStreamDeck::exists` (`f_stat`) from the engine's `prepare()` when the selector opens. `FatFile::open_write` creates `/tapes` on first record; single-digit slots keep the names 8.3-safe.

- **`ParamId::AltPos` + `CapAltPos`: a reusable Alt+POS knob layer.** Mirrors `CapAux`/`Aux` (Alt+PITCH): an engine that advertises `CapAltPos` receives the Alt+POS knob as `ParamId::AltPos` (the tape engine reads it as per-deck pan); engines without it keep POS -> `Pos` byte-for-byte, so granular/delay/edrums/passthrough are unchanged.

### Fixed

- **Granular slice fade-out at high pitch/speed (V6).** `Vox::_set_decay_start()` assigned the grain decay-start position only when the fade region was shorter than the slice; in the high-increment, small-slice corner (e.g. playhead increment 8.0 with the minimum slice of 1344 samples, where the fade region 1536 exceeds the slice) it left a stale, history-dependent value. It does not build or affect the firmware. Initial coverage locks the two findings that required manual tracing: `Divider` triplet timing does not drift over long runs (S1), and `SynClock` emits exactly `_ticks_per_clock` internal ticks per external clock pulse for 24->48 and 12->48 PPQN, including when the internal timer is over-driven (S3). Also covers the `Config` text parser (S5): valid parse, the `is_loaded()` default (V1), and the documented brittleness (mid-token whitespace stripping, unknown/overlong names ignored); the `wav.h` header builder/parser (canonical 44-byte layout, build->serialize->parse round-trip, rejection of short/non-RIFF input); and `SpeedMap` (semitone->speed table: octave/fifth anchors, end clamping, monotonicity). A `test/stubs/daisysp.h` shim supplies the few DaisySP symbols referenced by otherwise-pure headers (`fmap`, `fclamp`); and `Follower` (mean-square envelope follower: reset, steady-state convergence, output clamping, amp scaling, attack-faster-than-release); the `sample16` float<->int16 conversion (round-trip, clamping, quantization step); `convert_pcm_block` (the load shim: f32<->i16 block conversion, clamping, and float->i16->float round-trip within a quantization step); and `PcmLoader` - a simulation of the SD audio-load accounting (frame counts, truncation to capacity, termination, contiguous placement) across every width combo and many chunk sizes, exercising the exact code card.cpp runs. 116 checks across 9 classes.

- **`LOFI_INT16` 16-bit loop buffer (default off, opt-in via `make LOFI_INT16=1`).** Stores the loop buffer as 16-bit PCM instead of 32-bit float, **doubling record time to 84 s in the same SDRAM** (verified: SDRAM footprint identical at both depths; `SRAM_EXEC` +304 B for the conversion code). Pieces:

  - `src/core/sample16.h` - pure, host-tested float<->int16 convert with hard clip.

  - `Buffer::Frame` becomes `int16_t` under the flag (`Buffer::Sample` alias); conversion is confined to the two element-wise sites, `Buffer::_read` and `Buffer::write` (the overdub mix stays in float, then clamps + quantizes). Identity / byte-identical codegen when the flag is off.

  - `kSourceMaxSeconds` 42 -> 84 under the flag; `wav.h` emits a 16-bit PCM header; the SD load path (`card.cpp`) accepts the matching depth and computes frames from the correct byte width.

  - `Makefile` gains a `LOFI_INT16=1` build option (mirrors `DEBUG=1`). Audio storage behaviour in the default build is byte-identical (the `int16` paths compile out); the default save format is still 32-bit float.

- **Convert-on-load shim (`src/core/pcm_convert.h`, wired into `card.cpp`).** The SD load path now accepts BOTH 16-bit PCM and 32-bit float files in either build and converts to the buffer's storage width on the fly (`convert_pcm_block`, pure + host-tested). So a 16-bit firmware loads legacy float tapes and a float firmware loads 16-bit tapes - resolving the prior "rejects mismatched tapes" limitation; tapes are portable across builds. `kChunk` (32 KB) is a multiple of both sample widths and WAV data is sample-aligned, so chunks never straddle a sample (no carry buffer needed). Loading a longer tape than the buffer holds truncates to capacity. The streaming size/offset/ termination math lives in a pure, host-tested helper (`src/core/pcm_loader.h`, `PcmLoader`) that card.cpp delegates to, so the accounting is validated off-target rather than by reasoning alone. Note: this adds the dual-format acceptance + converter to the **default** build too (~+150 B `SRAM_EXEC`), so the default firmware now also reads 16-bit tapes - a small, additive behaviour change. The audible overdub-quantization character of the 16-bit buffer remains unverified by ear. See `docs/dev/lofi-int16-scope.md`.

- **Design scope docs** `docs/dev/lofi-path-b-scope.md` (half-rate buffer) and `docs/dev/lofi-int16-scope.md` (16-bit storage): inventories of the frame<->time/tempo coupling and the change sites for each lo-fi record-time approach.

### Fixed (portability)

- **`divider.h` now includes `<cmath>` for `std::round`** (was relying on `<math.h>`, which only places `round` in `std::` on the ARM toolchain, not in standard libc++). Surfaced by the new host harness; no effect on the firmware build. (`src/core/divider.h`)

- **`WavHeader::size` retyped `size_t` -> `uint32_t`.** The RIFF chunk-size field is a 32-bit field per the WAV spec; `size_t` is 4 bytes only on the 32-bit target, so the 44-byte layout (and its `static_assert`) held there but broke on a 64-bit host and would emit a malformed header on any 64-bit build. Layout-identical on-target (verified: the firmware binary is byte-for-byte unchanged in size); `wav.h` also now includes the `<cstdint>`/`<cstddef>`/`<cstring>` it uses rather than relying on its includers. Surfaced by the host harness. (`src/memory/wav.h`)

- **`follower.cpp` now includes `<cmath>` for `std::exp`** (was relying on `<math.h>`; same std-namespace portability issue as `divider.h`). Surfaced by the host harness; no effect on the firmware build. (`src/core/follower.cpp`)
